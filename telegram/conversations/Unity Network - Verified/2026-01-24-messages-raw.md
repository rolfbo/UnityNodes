Feature proposal:

This proposal replaces the default marketplace experience with a simple in-app licence claim flow designed for new users. On first use, a user either enters an optional referral code or skips it; if they skip, the app randomly assigns a licence from the available pool at a standardised 60% UNO / 40% ULO split. If they enter a valid referral code (one unique custom code per UNO), they claim a licence and receive a +10% rewards uplift, making the effective split 50% UNO / 50% ULO. Licences are effectively indefinite provided the user maintains at least 80% uptime, and the UNO can only revoke if uptime drops below that threshold. Separately, UNOs can still publish custom UNO/ULO splits that users claim manually using the longer codes we already use today, keeping advanced flexibility without forcing it into the default user journey.

It’s needed because the marketplace adds avoidable friction and decision fatigue at the exact point where non-technical users are most likely to churn. Standardising the default split eliminates race-to-the-bottom dynamics and prevents incentives from drifting towards short-term undercutting. A one-tap claim flow makes onboarding materially easier for people outside the existing crypto-native circles, while the referral uplift creates a clear, measurable incentive for UNOs to market and actively bring in new participants. In short, it improves conversion and retention by simplifying the path to first rewards, while preserving customisation for power users via the manual route.

*Caveat - __this request included an additional option to auto upgrade or downgrade ULOs through earning levels before they would have a license auto-revoked. There was also a more plain vanilla version that just requested having an auto revoke feature.__

— Submitted November 6, 2025 —

**Enable automatic tier upgrades and license revoking based on up-time requirements**
After an operator sets up several earning tiers (i.e. 30%, 50%, 70%, etc.) they will have the option to also tie these payment tiers to a required percentage of up-time for the licenses to be retained. 

Examples below:

__Tier 2
Earning potential: 50%
Up-time Requirement: at least 50% up-time for 60 days__

If the licensee falls below this requirement for a consecutive number of days (let’s say 5, this should be customizable) then the license will be auto-downgraded to the next lowest tier. If they continue to fall below the required up-time threshold for each tier they will eventually have their license revoked and they will have to request a new one (lowest tier).

Conversely, if they are meeting the up-time requirements for their current tier, they should be auto-upgraded to the next highest earning tier (example 80% for 60 days). As long as they are able to maintain a threshold of good uptime for the highest tier, they will be rewarded by being able to keep the license for perpetuity. 

In this scenario, all licenses are made available at the lowest tier first, and excellent uptime is rewarded by upgrades to a higher earning tier, and being able to retain the license indefinitely.
