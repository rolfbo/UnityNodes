Serhii Voloshyn, [Feb 7, 2026 at 08:41]
Here's why this problem arose and why static IP solved it:
1. The problem of "collective responsibility" (CGNAT)
Your previous address (100.64.107.114) was an internal address of the provider. This means that hundreds or even thousands of subscribers were accessing the Internet through the same common public IP address.
What happened: If someone else from your city (who is on the same provider) was doing suspicious things, sending spam, or simply updating the application too often, the server's security systems (Firewall/WAF) put this common IP address on the "blacklist".
Result: When you tried to connect, the server saw a "banned" address and gave you a 403 Forbidden, although you personally did not violate anything.
2. Specifics of the WebSocket protocol
Applications often use WebSocket for instant messages or updates. Unlike regular websites, WebSocket requires a long and stable connection.
Provider gateways (NAT) that serve dynamic addresses often "aggressively" terminate old connections to make room for new users.
The application server could view your router's attempt to break through this "double barrier" as an attack or a malformed request.

3. Why did it work before and then stop?
This could happen for three reasons:
Address pool change: The provider could have changed the routing, and you ended up in a subnet that was already in the spam filter database.
Application security update: Developers could have tightened security rules by prohibiting connections from addresses belonging to large provider networks (this is often done to combat bots).
Provider neighbor: Someone from your network segment began to show activity that activated server protection.
